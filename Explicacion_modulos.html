Explicación del Proyecto Carritos API

Módulos de Choferes, Carritos y Despachos — Estructura, punteros, errores, validaciones y mocks.

1. Estructura de los módulos (Choferes, Carritos, Despachos)

Cada módulo sigue el mismo patrón de 4 capas:

CapaUbicaciónResponsabilidad
Modelointernal/models/Define la estructura de datos y su validación
Storageinternal/storage/Acceso a SQLite (GORM). CRUD contra la base de datos
Handlerinternal/handlers/Recibe HTTP request, valida, llama al storage, responde JSON
Routerinternal/handlers/server.goRegistra las rutas HTTP en chi.NewRouter()

Ejemplo: Choferes

// Modelo — define qué es un chofer
type Chofer struct {
    ID       int          `gorm:"primaryKey"`
    Nombre   string       `gorm:"column:nombre_chofer"`
    Licencia string       `gorm:"column:licencia;uniqueIndex"`
    Celular  string       `gorm:"column:celular"`
    Estado   EstadoChofer `gorm:"column:estado_chofer"`
}

// Storage — implementa el CRUD en SQLite
func (a *AlmacenSQLite) GetChoferByID(id int) (Chofer, error) {
    var c Chofer
    if err := a.db.First(&c, id).Error; err != nil {
        return Chofer{}, ErrNotFound
    }
    return c, nil
}

// Handler — recibe HTTP, llama al storage, responde
func (h *ChoferHandler) GetByID(w http.ResponseWriter, r *http.Request) {
    id, _ := strconv.Atoi(chi.URLParam(r, "id"))
    chofer, err := h.almacen.GetChoferByID(id)
    if err != nil {
        RespondError(w, http.StatusNotFound, err.Error())
        return
    }
    RespondJSON(w, http.StatusOK, chofer)
}

2. Por qué y dónde se usan punteros

En Go, los punteros (& y *) se usan para modificar valores originales sin copiarlos.

Caso 1: GORM necesita punteros para escribir

var c Chofer
a.db.First(&c, id)    // ← le pasamos &c para que GORM escriba EN c
internal/storage/sqlite_choferes.go:20

Sin el &, GORM crearía una copia y c quedaría vacía.

Caso 2: Métodos receptores con puntero

func (a *AlmacenSQLite) GetChoferByID(...) { ... }  // ← a es puntero
internal/storage/sqlite_choferes.go:18

Si el receptor fuera a AlmacenSQLite (sin *), cada método recibiría una copia del struct y no podría acceder a la conexión de BD real.

Caso 3: El campo Chofer del modelo Solicitud

type Solicitud struct {
    Chofer *string `json:"chofer"`  // ← puntero: puede ser nil (sin chofer asignado)
}
internal/models/Solicitudes.go:9
Regla simple:
& antes de una variable = "dame la dirección de memoria"
* en tipo de función = "este argumento se modifica afuera"
* en tipo de campo = "puede ser nil"

3. Cómo se manejan recursos no encontrados

El patrón usa storage.ErrNotFound + errors.Is() en el handler.

Paso 1 — Storage define el error

// internal/storage/errores.go
var ErrNotFound = errors.New("recurso no encontrado")
internal/storage/errores.go:5

Paso 2 — Storage lo retorna cuando GORM dice "record not found"

func (a *AlmacenSQLite) GetChoferByID(id int) (Chofer, error) {
    var c Chofer
    if err := a.db.First(&c, id).Error; err != nil {
        if isNotFoundError(err) {  // detecta el error de GORM
            return Chofer{}, ErrNotFound  // lo traduce a nuestro error
        }
        return Chofer{}, err
    }
    return c, nil
}
internal/storage/sqlite_choferes.go:18-27
if err := h.almacen.GetChoferByID(id); err != nil {
    if errors.Is(err, storage.ErrNotFound) {
        RespondError(w, http.StatusNotFound, err.Error())  // ← 404
        return
    }
    RespondError(w, http.StatusInternalServerError, err.Error())  // ← 500
}

Flujo completo: GORM "record not found" → isNotFoundError() lo detecta → se retorna storage.ErrNotFound → errors.Is() lo identifica → HTTP 404.

6. Línea al azar explicada

_ = json.NewEncoder(w).Encode(payload)
internal/handlers/responses.go:13

Esta línea está en responses.go:13. Se descompone en 3 partes:

ParteQué hace
json.NewEncoder(w)Crea un codificador JSON que escribe directamente en el response writer w
.Encode(payload)Convierte payload a JSON y lo escribe en w
_ = ...Descarta el error (porque si el cliente ya se desconectó, no hay nada que hacer)

¿Por qué no usamos json.Marshal?

// Forma lenta (asigna bytes en memoria):
datos, _ := json.Marshal(payload)
w.Write(datos)

// Forma eficiente (escribe directo al stream):
json.NewEncoder(w).Encode(payload)

7. Qué pasa si quito el asterisco (*)

Caso 1: Receptor del método

// CON puntero — funciona:
func (a *AlmacenSQLite) GetChoferByID(id int) (Chofer, error) {
    a.db.First(&c, id)  // usa la conexión real
}

// SIN puntero — ERROR de compilación:
func (a AlmacenSQLite) GetChoferByID(id int) (Chofer, error) {
    a.db.First(&c, id)  // a.db es una COPIA, no la conexión original
}
// El método no puede modificar ni acceder al estado real de "a"
internal/storage/sqlite_choferes.go:18 — receptor con puntero vs sin puntero
Resultado: Go te da un error de compilación o el código compila pero a.db apunta a una copia del puntero, no a la conexión real de SQLite. Los datos nunca se guardarían.

Caso 2: Campo Chofer en Solicitud

// CON puntero — puede ser nil (sin chofer):
Chofer *string `json:"chofer"`

// SIN puntero — siempre tiene un valor (string vacío ""):
Chofer string `json:"chofer"`
internal/models/Solicitudes.go:9 — campo puntero vs campo valor

Si quitas el *, no puedes distinguir entre "sin chofer" y "chofer vacío".

8. Qué es defer y dónde se usa

defer ejecuta una función al finalizar la función actual, sin importar si hubo error o return.

Ejemplo en el proyecto (storage/factory.go)

func Inicializar() Recursos {
    db := abrirGorm()

    return Recursos{
        Almacen: NuevoAlmacenSQLite(db),
        Cleanup: func() {
            sqlDB, _ := db.DB()
            sqlDB.Close()
        },
    }
}

// En main.go se llama defer para cerrar al salir:
recursos := Inicializar()
defer recursos.Cleanup()  // se ejecuta cuando main() termina
internal/storage/factory.go:15-25 — Inicializar()  |  cmd/Carritos-api/main.go:50-55 — defer Cleanup()

Ejemplo genérico

func procesar() {
    archivo := abrirArchivo()
    defer archivo.Close()  // siempre se cierra, pase lo que pase

    // ... procesar archivo ...
    if error {
        return  // defer se ejecuta A PESAR del return
    }
}
Ejemplo genérico — patrón común de defer
Regla: defer se ejecuta cuando la función retorna, en orden LIFO (el último defer se ejecuta primero).

9. Tutorial: Mock que retorna error → handler responde 500

Vamos a hacer un cambio manual paso a paso:

Paso 1 — Abre el archivo de test

internal/handlers/choferes_handler_test.go

Paso 2 — Agrega este test después de los existentes

func TestChoferHandlerGetByIDErrorServidor(t *testing.T) {
    r, mockRepo := setupChoferHandlerTest(t)

    // Configurar el mock para que retorne un error de BD
    mockRepo.On("GetChoferByID", 50).
        Return(models.Chofer{}, errors.New("error de base de datos")).Once()

    // Hacer request al endpoint
    req := httptest.NewRequest("GET", "/api/v1/choferes/50", nil)
    rr := httptest.NewRecorder()
    r.ServeHTTP(rr, req)

    // Verificar que el handler respondió 500
    require.Equal(t, http.StatusInternalServerError, rr.Code)

    // Verificar que el body contiene el error
    var respuesta map[string]string
    json.NewDecoder(rr.Body).Decode(&respuesta)
    assert.Equal(t, "error de base de datos", respuesta["error"])

    // Verificar que el mock fue llamado
    mockRepo.AssertExpectations(t)
}
